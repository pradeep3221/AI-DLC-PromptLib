## AI Guardrails (Anti-Hallucination Rules)

**These rules are MANDATORY for every prompt in this library.**

### Rule 1: Verify Before Listing

```
❌ NEVER list a file, component, or dependency without verifying it exists
✅ ALWAYS use directory listing / file search to confirm existence first
```

- Before listing ANY component in an inventory → verify the file/folder exists
- Record the actual path found (not assumed)
- If a file doesn't exist → do NOT include it in output

### Rule 2: Metrics Must Come From Actual Tool Output

```
❌ NEVER fabricate or estimate metrics (line counts, complexity scores, coverage %)
✅ ALWAYS derive metrics from actual analysis tool output or code inspection
```

- Code complexity → must come from static analysis or manual count
- Test coverage → must come from test runner output
- Line counts → must come from actual file inspection
- Performance metrics → must come from actual monitoring/APM data

### Rule 3: Acknowledge Uncertainty

```
❌ NEVER present assumptions as facts
✅ ALWAYS flag uncertain findings with confidence levels
```

- Use the [Confidence Level Framework](#confidence-level-framework) for all extracted rules
- When information is incomplete → say so explicitly
- When multiple interpretations exist → list all with reasoning

### Rule 4: Evidence Required

```
❌ NEVER state a business rule without evidence
✅ ALWAYS include: file path, line numbers, code snippet
```

- Every extracted rule must reference source location
- Every claimed dependency must show the reference code
- Every metric must cite the measurement source

### Rule 5: Cross-Reference Before Promoting Confidence

```
❌ NEVER mark a rule as HIGH confidence without supporting evidence
✅ ALWAYS search for tests, docs, and comments before assigning confidence
```

## Execution Workflow

### Step 1: Classify the Project (ALWAYS FIRST)

Run **Prompt P0** from [phase0-classification.md](phase0-classification.md):
- Determine project intent (Enhancement / Modernization / Migration)
- Determine architecture type (Monolith / Distributed / EDA / Hybrid)
- Generate `project-profile.yaml`

### Step 2: Select Prompt Depth

Based on classification, use the [Depth Calibration Matrix](#depth-calibration-by-project-type) to determine which prompts to run and at what depth.

### Step 3: Execute Phases 1–3 (Inventory, Code Health, Architecture)

```
P1.1 Component Inventory
P1.2 Technology Matrix
P1.3 Dependency Mapping
P1.4 Domain Model (if Modernization/Re-architect)
P1.5 Bounded Contexts (if Modernization/Re-architect)
P1.6 Performance Baselining (if Migration)
P2.1–P2.4 Code Health Assessment
P3.1–P3.6 Architecture Scoring (if Modernization/Re-architect)
```

### Step 4: Execute Phases 4–6 (Domain Discovery)

```
P4.1–P4.7   Core Discovery (matching technologies)
P5.1–P5.10  Technology-Specific Extraction (matching stack)
P6.1–P6.12  Legacy Deep Extraction (as needed)
```

### Step 5: Consolidate & Score

- Run scoring engine (Prompt P3.4)
- Generate migration readiness (Prompt P3.5)
- Produce portfolio view if multi-project (Prompt P3.6)

## Output Standards

### File Naming Convention

```
/docs/{phase}/{prompt-id}-{short-name}.md
```

Example:
```
/docs/inventory/P1.1-component-catalog.md
/docs/health/P2.1-dependency-graph-analysis.md
/docs/domain/P4.6-sp-business-logic-extraction.md
```

### Markdown Structure (Every Output File)

```markdown
# {Title}

**Generated By**: Prompt {ID} — {Name}
**Date**: {ISO date}
**Input**: {Source files/prompts used}
**Confidence**: {Overall confidence: HIGH/MEDIUM/LOW}

---

## Summary
{2-3 sentence executive summary}

## Findings
{Main content with tables, diagrams, code snippets}

## Risks & Gaps
{Identified risks, missing information, uncertain areas}

## Next Steps
{What to do with this output — feeds into which prompt}
```

### Diagram Standards

- Use **Mermaid** syntax for all diagrams (dependency graphs, flowcharts, state machines)
- Use **ASCII** as fallback when Mermaid isn't supported
- Include coupling metrics (Ca, Ce, Instability) on dependency nodes

### Table Standards

- Every inventory/catalog uses tables with consistent columns
- Include a "Confidence" or "Risk" column where applicable
- Sort by priority/severity (highest first)

## Confidence Level Framework

**Every extracted business rule, dependency, or finding MUST include a confidence level.**

| Level | Criteria | Evidence Required | Action |
|-------|----------|-------------------|--------|
| **HIGH** | Rule has explicit code comment explaining intent, OR has corresponding unit test, OR matches documented requirements | Comment text, test file path, or requirements doc reference | ✅ Can proceed with migration |
| **MEDIUM** | Rule is clear from code logic (explicit IF/CASE with business values) but no documentation or tests found | Code snippet with line numbers showing clear conditional | ⚠️ Flag for review before migration |
| **LOW** | Rule is inferred from naming conventions, spread across multiple locations, or logic is ambiguous | Multiple file references, explanation of inference | ❌ MUST have human validation |
| **UNCERTAIN** | Cannot determine if code is a business rule or technical implementation | Code snippet with explanation of ambiguity | ❌ MUST NOT migrate without explicit human sign-off |

### Confidence Promotion Rules

| Found | Promotion |
|-------|-----------|
| Matching unit test found | LOW/MEDIUM → promote one level |
| Matching documentation found | LOW/MEDIUM → promote one level |
| Both test AND docs found | Any → HIGH |
| Conflicting implementations found | Any → demote to UNCERTAIN |

## Validation Checkpoints

Run these checks at the end of each phase:

### Phase 0 Checkpoint

```markdown
### Phase 0 Validation
- [ ] All inventoried components verified to exist (file paths confirmed)
- [ ] Technology matrix covers all discovered tech (no gaps)
- [ ] Dependency map has no orphan nodes
- [ ] Performance baseline captured for critical paths
- [ ] Code health metrics derived from actual analysis (not estimated)
- [ ] Scoring model populated with actual data
```

### Phase 1 Checkpoint

```markdown
### Phase 1 Validation
- [ ] Every extracted business rule includes confidence level
- [ ] All code snippets include exact file path and line numbers
- [ ] HIGH confidence rules have supporting evidence documented
- [ ] MEDIUM/LOW/UNCERTAIN rules have human review action items
- [ ] Cross-reference with tests completed
- [ ] Cross-reference with documentation completed
- [ ] No rules presented without evidence
- [ ] Ambiguous patterns flagged (not interpreted)
```
