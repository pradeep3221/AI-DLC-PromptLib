## P0.1 Determine Project Intent

```
**Goal**: Classify the primary intent driving this engagement

**Classify into ONE of the following**:

| Intent | Code | Description | Key Indicator |
|--------|------|-------------|---------------|
| Brownfield – Enhancement | BF-ENH | Adding features to existing system | No architecture changes planned |
| Brownfield – Modernization | BF-MOD | Improving architecture while keeping system alive | Strangler pattern, incremental improvement |
| Migration – Rehost | MIG-RH | Lift & shift to new infrastructure | Same code, different host |
| Migration – Replatform | MIG-RP | Move to new platform with minor code changes | .NET Framework → .NET 8, IIS → containers |
| Migration – Re-architect | MIG-RA | Full redesign and rebuild | New architecture, new services |

**Decision Tree**:

1. Is the goal to add new features without changing architecture?
   → YES = BF-ENH
   
2. Is the goal to improve the existing codebase incrementally?
   → YES = BF-MOD
   
3. Is the goal to move to new infra with no/minimal code changes?
   → YES = MIG-RH
   
4. Is the goal to upgrade the platform (.NET version, hosting)?
   → YES = MIG-RP
   
5. Is the goal to redesign from scratch using modern patterns?
   → YES = MIG-RA

**Output**: Project intent classification with rationale
```

## P0.2 Determine Architecture Type

```
**Goal**: Classify the CURRENT application architecture

**Scan indicators**:
- Solution structure (*.sln, *.csproj count and types)
- Communication patterns (REST, gRPC, message queues)
- Deployment model (single deploy vs. multiple services)
- Data model (single DB vs. DB per service)

**Classify into the most fitting type**:

| Architecture | Code | Indicators |
|-------------|------|------------|
| Monolith | MONO | Single solution, single deployment, shared DB |
| API + SPA | API-SPA | Backend API + separate frontend (Angular/React/Vue) |
| Worker / Background | WORKER | Console apps, Windows Services, scheduled tasks |
| Console / Batch | BATCH | Batch processing, ETL jobs, CLI tools |
| Distributed (Microservices) | DIST | Multiple independent services, separate deployments |
| Event-Driven Architecture | EDA | Message brokers, pub/sub, event sourcing |
| Hybrid | HYBRID | Mix: API + Events + Workers + Monolith components |

**Secondary Classification** (can have multiple):
- Sync-heavy (REST/gRPC dominant)
- Async-heavy (message queues dominant)
- Data-centric (heavy stored procedure logic)
- UI-heavy (complex frontend with embedded business logic)

**Output**: Architecture classification with evidence
```

```
**Goal**: Generate a standardized project-profile.yaml as the single source of truth

**Output**: `/project-profile.yaml`

**Template**:

  name: "{Project Name}"
  intent: "{BF-ENH | BF-MOD | MIG-RH | MIG-RP | MIG-RA}"
  intent_description: "{One-line rationale}"
  owner: "{Team/person}"
  start_date: "{ISO-date}"

  current_type: "{MONO | API-SPA | WORKER | BATCH | DIST | EDA | HYBRID}"
  current_style: "{Sync-heavy | Async-heavy | Data-centric | UI-heavy}"
  target_type: "{Target architecture, if known}"
  deployment_model: "{IIS | Azure App Service | AKS | Containers | Windows Service}"

  current:
    dotnet: "{.NET Framework 4.x | .NET Core 3.1 | .NET 6 | etc.}"

  total_count: 0
  solutions: []     # List of .sln files
  projects: []      # List of .csproj/.vbproj with type
  services: []      # Runtime services
  databases: []     # Database instances

  sync: []          # REST, gRPC, WCF, ASMX
  async: []         # Service Bus, MSMQ, RabbitMQ
  file_based: []    # SFTP, file drops

data:
  databases:
    - name: ""
      type: "{SQL Server | Oracle | MongoDB | Redis}"

  shared_db: false  # Any shared databases?

  size: 0
  familiarity_with_target: "{High | Medium | Low}"

  timeline: ""
  budget: ""

**Instructions**:
- Fill ALL fields you can determine from scanning the codebase
- Mark unknown fields with `"UNKNOWN — verify with team"`
- This YAML feeds into scoring (P3.4) and depth calibration

## P0.4 Assessment Depth Matrix

```
**Goal**: Based on the project classification, determine which prompts to run and at what depth

**Use this matrix after completing P0.1 and P0.2**:

| Prompt Area | BF-ENH | BF-MOD | MIG-RH | MIG-RP | MIG-RA |
|-------------|:------:|:------:|:------:|:------:|:------:|
| **Phase 1: Inventory & Baseline** |
| P1.1 Inventory | Light | Full | Full | Full | Full |
| P1.2 Tech Matrix | Light | Full | Full | Full | Full |
| P1.3 Dependencies | Light | Full | Light | Moderate | Full |
| P1.4 Domain Model | Skip | Full | Skip | Skip | Full |
| P1.5 Bounded Contexts | Skip | Full | Skip | Skip | Full |
| P1.6 Perf Baseline | Skip | Full | Full | Full | Full |
| **Phase 2: Code Health** |
| P2.1–P2.4 Code Health | Hotspots | Full | Skip | Moderate | Full |
| **Phase 3: Architecture & Scoring** |
| P3.1 Multi-Layer | Skip | Full | Skip | Light | Full |
| P3.2 Event Maturity | Skip | If EDA | Skip | If EDA | Full |
| P3.3 Triangulation | Skip | Full | Skip | Skip | Full |
| P3.4 Scoring | Skip | Full | Light | Moderate | Full |
| P3.5 Migration Ready | Skip | Full | Full | Full | Full |
| P3.6 Portfolio View | Skip | If multi | If multi | If multi | Full |
| **Phase 4–6: Discovery** |
| P4.1–P4.7 Core | Hotspots | Full | Skip | Light | Full |
| P5.1–P5.10 Tech-Specific | Matching | Matching | Skip | Matching | All |
| P6.1–P6.12 Legacy Deep | Skip | Full | Skip | Key areas | Full |

**Depth Definitions**:
- **Full**: Execute all sub-prompts, capture all outputs
- **Moderate**: Execute primary sub-prompts, skip detailed breakdowns
- **Light**: Execute summary-level only, focus on key findings
- **Hotspots**: Only analyze high-risk / high-change areas
- **Matching**: Only run prompts matching the technology stack found
- **Skip**: Do not execute (not relevant for this project type)

**Output**: `/docs/inventory/assessment-plan.md`
```

## P0.5 Project Type Decision Guide

```
**Goal**: Help teams who aren't sure which project type they are

**Answer these 10 questions** (Yes/No):

| # | Question | If Yes → Points Toward |
|---|----------|----------------------|
| 1 | Is the existing system staying in production as-is? | BF-ENH |
| 2 | Are you adding features without changing architecture? | BF-ENH |
| 3 | Are you improving code quality while keeping the system alive? | BF-MOD |
| 4 | Are you introducing DDD / Clean Architecture patterns? | BF-MOD |
| 5 | Are you using the Strangler Fig pattern? | BF-MOD |
| 6 | Are you moving to new servers/cloud with no code changes? | MIG-RH |
| 7 | Are you upgrading .NET version or hosting platform? | MIG-RP |
| 8 | Are you extracting config/secrets for cloud deployment? | MIG-RP |
| 9 | Are you breaking a monolith into microservices? | MIG-RA |
| 10 | Are you redesigning domain boundaries and data ownership? | MIG-RA |

**Scoring**:
- Highest count → primary classification
- If tied → choose the MORE ambitious classification (higher effort = safer to over-assess)

**Edge Cases**:
- "We're rehosting BUT also upgrading .NET" → MIG-RP (replatform includes rehost)
- "We're modernizing BUT also breaking into services" → MIG-RA (re-architect includes modernization)
- "We're just fixing bugs and adding a feature" → BF-ENH (enhancement scope)

**Output**: Final classification with rationale, stored in `project-profile.yaml`
```
